# Store-and-Forward (서버 송신 버퍼링) 사양 (v1.0)

> 대상: 미니 PC ↔ 서버(web) MQTT 송신 구간 · 전 화면 공통 상태바
> 기획 버전: ERUT_ServiceFlow_FixedProbe.html v31.0
> 작성일: 2026-06-24
> 컴포넌트: `window.StatusBar` (Chrome.jsx, `mqttPending` prop) · `window.NotificationCenter` (Chrome.jsx) · [4-2] MQTT 설정 화면

---

## ⚠️ 개발팀 필독 — 프로토타입 미시연 항목

**이 기능은 현재 프로토타입에서 동작이 시각적으로 시연되지 않습니다.** 백엔드 구현 + 라이브 배선이 필요한 항목입니다.

- 상태바 컴포넌트(`StatusBar`)에 `mqttPending` prop과 렌더 로직은 추가되어 있으나, `index.html`에서 **`mqttPending={0}` 고정**으로 전달되어 평소엔 '연결됨'만 보입니다.
- 끊김/재송신을 실제로 보이게 하려면 ① 미니 PC의 로컬 송신 큐(미전송 카운트) 집계 ② MQTT 연결 상태 감지 ③ 그 값을 `mqttPending` / `mqttConnected`에 실시간 반영하는 배선이 필요합니다.
- 상태 3종(연결됨 / 끊김·송신 대기 / 송신 중)의 **정적 디자인은 [4-2] MQTT 설정 화면에 데모 블록으로 문서화**되어 있습니다.

## 1. 배경 / 결정

고정형은 설치 후 **상시 연속 측정** 상태다. 현장은 Wi-Fi/LTE가 불안정하므로, 서버(web) 연결이 끊긴 동안 측정 데이터가 유실되면 감육 트렌드에 공백이 생긴다. 이를 막기 위해 **store-and-forward**(로컬 보관 → 재연결 시 자동 재송신)를 도입한다.

**사용자 결정 (3건)**
- **적용 구간**: 미니 PC ↔ 서버(web) MQTT 송신 구간 **한정**. (MC ↔ 미니 PC 끊김은 측정 데이터가 미니 PC로 유입되지 않아 버퍼 대상이 아님 → '측정 중단' 이벤트로 별도 처리.)
- **UI**: 상태바 MQTT pill 확장 + 알림 센터 연동. (상세 백로그 패널은 도입하지 않음.)
- **버퍼 한계(디스크 포화) 처리**: 이번 범위 제외 — 후속 결정.

**제품 분리 관점**: 데이터 유실 방지 = 현장 수집 신뢰성 → **윈도우 책임**에 부합 (web 트렌드/판정과 무관).

---

## 2. 동작 흐름

1. 미니 PC는 측정 데이터를 **항상 로컬에 먼저 저장**한다 (SQLite WAL 메타 + `.bin` 파일, 기존 아키텍처). 송신 성공 여부와 무관하게 로컬 저장은 항상 수행.
2. 각 데이터 레코드는 **송신 상태 플래그**를 가진다: `pending`(미전송) / `sent`(전송 완료).
3. 서버(web) MQTT 연결이 끊기면 → 신규 측정 레코드는 `pending`으로 로컬에 누적된다. 측정·로컬 저장은 정상 지속.
4. 재연결되면 → `pending` 레코드를 **가장 오래된 것부터 순서대로(FIFO) 자동 재송신**(catch-up)하여 미전송 0건까지 따라잡는다.
5. 재송신 완료 시 알림 센터에 정보(info) 알림 1건을 남긴다.

> 순서 보장(FIFO)은 시계열 트렌드 정합을 위해 필수. QoS 1 + 멱등 처리(중복 도착 시 서버측 dedup) 권장 — 기존 `Error_Code_Spec_v1.0.md` E703(QoS 1 ACK 누락 재발행)과 정합.

---

## 3. 데이터 모델 (제안)

송신 큐는 로컬 보관 레코드의 전송 상태로 표현한다. 별도 큐 테이블 또는 측정 레코드의 컬럼 중 택1.

| 필드 | 타입 | 설명 |
|---|---|---|
| `sendState` | enum | `pending` / `sent` |
| `enqueuedAt` | datetime | 로컬 보관(측정) 시각 — FIFO 정렬 키 |
| `sentAt` | datetime\|null | 송신 완료 시각 (재송신 포함) |
| `retryCount` | int | 재발행 횟수 (E703 연동) |

- **미전송 카운트** = `count(sendState == 'pending')` → 상태바 `mqttPending`에 반영.
- **가장 오래된 미전송 시각** = `min(enqueuedAt where pending)` (향후 백로그 표시용, 이번 UI 미사용).

## 4. UI — 상태바 MQTT 표시

`window.StatusBar`의 MQTT 그룹: `MQTT : [pill]` 옆에 미전송 누적 시에만 텍스트 라벨을 노출한다 (별도 pill 아님, 좌측 MQTT 그룹에 인접).

| 상태 | pill | 보조 텍스트 | 색 |
|---|---|---|---|
| 정상 송신 | `연결됨` (green LED) | — | — |
| 끊김 + 미전송 | `끊김` (red LED) | `송신 대기 N건` | `--system-caution` |
| 재연결 catch-up | `연결됨` (green LED) | `송신 중 N건` | `--system-info` |

- `mqttPending == 0`이면 보조 텍스트 미표시.
- 표시 형태 = **상태바 텍스트 라벨**(`erut-statusbar__text`), pill로 승격하지 않음 (사용자 결정, 2026-06-24).
- 구현 위치: `Chrome.jsx` `StatusBar({ mqttConnected, mqttPending })`. `mqttConnected` false + `mqttPending>0` → '송신 대기', `mqttConnected` true + `mqttPending>0` → '송신 중'.

## 5. UI — 알림 센터

| 시점 | severity | title | detail |
|---|---|---|---|
| 재연결 + 재송신 완료 | info | 서버 재연결 · 미전송 송신 완료 | 서버 통신 복구 (IP) · 끊김 동안 로컬 보관한 미전송 데이터 자동 송신 완료 |

- 현재 `MOCK.notifications`의 통신 복구 알림(n5)을 위 내용으로 보강 적용.
- (선택) 끊김 발생 시점에 경고(caution) 알림을 추가할 수 있으나 이번 범위에선 상태바 표시로 갈음.

---

## 6. 범위 밖 (후속 결정 필요)

- **버퍼 한계 / 디스크 포화**: 장기 미연결로 `pending`이 누적되어 디스크가 찰 때의 처리(경고 배너 · 순환 삭제 · 송신 우선순위). 미정.
- **상세 백로그 패널**: [4-2]에 미전송 건수·버퍼 사용량·재송신 진행률을 보여주는 패널 — 이번 미도입(상태바+알림으로 충분 판단).
- **버퍼 임계 에러코드**: `Error_Code_Spec`에 송신 큐 적체 임계용 코드 추가 검토 (현재 E604 미니PC 처리 큐 적체 / E703 QoS ACK 누락으로 일부 커버).
